## Context

- 已有 `dish` / `dish_alias` / `food_item`（标准食物）目录模型与 AI 由后端统一调用的治理要求。
- DashScope 文本链路已可通；本能力需接入**多模态**（Qwen-VL 系列），并在服务端完成词表拼装、图片预处理与结果校验。
- 用户确认：MVP **整表词表一次性进 prompt**（当前数据量可接受）；**别名字符串 + 服务端映射**；**τ = 0.7**；**同步 API**；**先压缩再送模型**且尽量不伤识别；响应含 **`request_id`** 与拒识原因；边界场景**明确错误码**。

## Goals / Non-Goals

**Goals:**

- 单图识别：优先唯一、高置信度 `dish`；否则输出最多 10 条带置信度的库内食材候选。
- 模型输出严格 JSON；非 JSON 或 schema 不通过 → 可重试或错误码（见 spec）。
- Prompt 明确**非食物**、**无法识别**时的模型行为，并由服务端映射为统一错误码。
- Id 类型与库一致（BIGINT），对外 JSON 使用**十进制字符串**。

**Non-Goals:**

- 向量库、按门店/菜单分片词表、异步任务队列（后续变更）。
- 修改 OpenSpec 中已有 `meal-*` 规范的条文（本变更仅新增能力 spec）。

## Decisions

1. **预处理**：先读图 → 若超过约定最大边长/体积则**等比缩小**（优先保最长边上限，避免过度 JPEG 重压缩）；仅在必要时做有损压缩，参数保守，并记录原图与处理后分辨率于内部日志（可选进 `model_meta`）。
2. **模型**：选用 DashScope Qwen-VL（具体 `model` 由配置指定，与现网百炼开通一致）；单次同步调用。
3. **词表注入**：prompt 内含**全量** `dish` 展示名、`dish_alias` 字符串、`food_item` 名称（及用于映射的主键 id 字符串）；模型对菜返回**别名字符串或标准名**（落在词表内），服务端查表得 `dish_id`。
4. **置信度**：由模型在 JSON 中自报；服务端以 **0.7** 为默认阈值（可配置），分别约束「整道菜」与「每条食材」（见 spec）。
5. **多别名命中同一菜**：若多条别名记录匹配同一 `dish_id`，在 `match.alias_id` 中取**确定性第一条**（例如按 `dish_alias` 主键升序），避免随机性。
6. **dish 与 ingredients 共存**：允许模型同时返回；**服务端**负责业务裁决（用户确认由后续餐次流程处理）。
7. **错误模型**：业务可恢复/可预期边界（非食物、无法识别、模型输出非法）使用**明确错误码**；基础设施失败单独分类（见 spec）。

## Risks / Trade-offs

- **[Risk] 词表增长导致 prompt 变长、成本上升** → Mitigation：监控 token；后续变更引入分片或向量检索。
- **[Risk] 模型自报置信度校准差** → Mitigation：线上按菜品/食材类型调参；必要时增加二次判别或人工确认流（已有识别过程表可承接）。
- **[Risk] 压缩损伤细粒度 OCR/小菜识别** → Mitigation：以分辨率上限为主、质量参数保守；对失败样本回传原图尺寸统计以调参。

## Migration Plan

- 新 API 与配置项：**新增**即可，无数据迁移。
- 回滚：关闭路由或 feature flag（若引入）、还原配置，不影响已有数据。
