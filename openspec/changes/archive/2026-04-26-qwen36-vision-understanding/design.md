## Context

当前项目已具备基础 AI 文本能力与餐食记录流程，但图片识别路径仍以 stub 为主，无法在生产环境提供可用的菜品/食材识别结果。此次变更需要在 `services/api` 中新增视觉理解能力，并与现有餐食记录链路对齐：输入图片（或已上传文件标识），调用 DashScope 兼容接口上的 `qwen3.6-flash`，输出结构化识别结果与可追踪错误码。

约束包括：  
- 需要保持现有 API 风格和统一错误处理机制。  
- 模型调用依赖外部服务，必须控制超时并避免将敏感配置写入代码库。  
- 识别结果需可被客户端稳定消费，避免自由文本导致解析不确定性。

## Goals / Non-Goals

**Goals:**
- 提供最小可用的视觉理解后端接口，默认模型为 `qwen3.6-flash`。
- 定义稳定的请求/响应结构，支持菜品与食材识别结果返回。
- 建立配置化模型客户端与错误映射，覆盖常见上游失败场景。
- 提供端到端验证路径（集成测试或可替代测试）确保变更可回归。

**Non-Goals:**
- 不在本次变更中实现复杂的多模型路由或自动降级策略。
- 不实现完整营养计算与长期识别结果持久化方案。
- 不在本次设计中确定前端最终交互细节（仅约束接口契约与失败语义）。

## Decisions

1. **采用独立 AI 视觉模块分层（Controller / Service / Client）**  
   - 决策：在 `modules/ai` 内新增视觉理解专用入口与服务，而非直接复用餐食模块控制器。  
   - 原因：隔离外部模型依赖，便于后续扩展不同视觉能力；也可复用既有 AI 错误处理抽象。  
   - 备选：直接耦合到餐食记录控制器。未采用原因是会放大控制器职责并增加测试复杂度。

2. **请求模型输出使用“受控结构化文本 + 服务端解析”**  
   - 决策：通过固定 prompt 约束 `qwen3.6-flash` 输出 JSON 结构，服务端进行严格解析与校验后再返回。  
   - 原因：模型原生自由文本不稳定，解析前置可提升客户端契约稳定性。  
   - 备选：直接透传模型原文。未采用原因是客户端兼容成本高且难以治理异常输出。

3. **配置驱动的模型参数与连接参数**  
   - 决策：将 API Key、base URL、model、timeout、重试策略放入配置类，并通过环境变量注入。  
   - 原因：支持多环境部署与安全管理，避免硬编码。  
   - 备选：仅在代码常量中声明模型名。未采用原因是无法灵活切换环境策略。

4. **统一错误码映射到业务异常层**  
   - 决策：将鉴权失败、限流、超时、上游 5xx、响应格式错误映射为内部 `AiErrorCode` 与统一 HTTP 响应体。  
   - 原因：客户端需要可预测的失败语义用于提示与回退。  
   - 备选：保留上游原始错误。未采用原因是耦合供应商细节且不稳定。

## Risks / Trade-offs

- **[模型响应不稳定]** → 通过严格 prompt、JSON schema 校验与解析失败降级错误码降低风险。  
- **[外部依赖可用性波动]** → 设置超时、有限重试与可观测日志；必要时让调用快速失败并提示重试。  
- **[误识别影响用户信任]** → 响应中保留置信度与原始候选信息（在可控范围内），交由客户端进行确认交互。  
- **[成本与延迟增加]** → 默认使用 `qwen3.6-flash` 以平衡性能/成本，并限制输入尺寸与并发策略。  

## Migration Plan

1. 增加配置模板与运行时环境变量（本地、测试、生产）。  
2. 先在 API 层上线灰度可访问的视觉识别端点，不替换现有客户端默认路径。  
3. 客户端按开关切换至新接口；保留 stub 回退路径用于紧急兜底。  
4. 监控错误码与延迟指标，确认稳定后再扩大流量。  
5. 若发生严重回归，关闭开关并回退到旧识别路径（stub/旧接口）。
