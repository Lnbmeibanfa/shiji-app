## Context

当前 `services/api` 处于后端骨架阶段，尚无可用用户认证实现，仅具备 Spring Boot + JPA + Security 基础依赖与数据库/Redis连接配置。移动端即将进入登录联调阶段，需要先建立可用、可测试、可演进的认证基础能力，并与既有数据库表（`user`、`user_auth`、`user_session`、`sms_code_log`、`user_agreement_accept_log`）对齐。

本次设计受以下已确认业务约束驱动：
- 验证码有效期 1 分钟；
- 验证码错误最大次数 10 次；
- 会话有效期 30 天；
- 单设备策略；
- 未同意协议时禁止登录；
- 退出登录执行“全部退出”。

短信供应商尚未确定，因此短信网关采用可替换适配接口，先完成业务闭环与可测试性。

## Goals / Non-Goals

**Goals:**
- 提供短信验证码发送与登录能力，形成完整登录闭环。
- 提供 token 会话恢复与全部退出能力，满足单设备会话策略。
- 在登录前强制校验协议同意状态，并将同意行为留痕。
- 通过分层设计与明确状态机，使认证流程可测且易于扩展（后续第三方登录可接入 `user_auth`）。

**Non-Goals:**
- 不在本次接入真实短信供应商，只预留发送适配层。
- 不在本次实现多因素认证、风控画像、设备指纹高级策略。
- 不在本次实现后台管理端的登录审计查询页面。

## Decisions

### 1) 模块边界：新增 `modules/auth`，与 `modules/user` 解耦
- 决策：认证相关 Controller/Service/Repository 集中在 `modules/auth`，用户资料仍保留在 `modules/user`。
- 原因：登录流程涉及验证码、会话、协议留痕等跨实体编排，独立模块更利于后续扩展（如微信/Apple 登录）。
- 备选方案：
  - 放入 `modules/user`：短期路径更短，但会使用户模块承担过多认证职责，不利于演进。

### 2) 会话令牌：服务端会话表 + token 哈希存储
- 决策：客户端持有随机 token；服务端在 `user_session` 存 `token_hash`，请求时按 hash 匹配。
- 原因：避免明文 token 落库泄露风险，支持会话状态控制（失效、过期、退出）。
- 备选方案：
  - 纯 JWT 无状态：实现快，但“全部退出/单设备互斥”控制复杂，且撤销成本高。

### 3) 单设备策略：登录成功后失效该用户其他有效会话
- 决策：验证码登录成功后，当前用户历史有效会话统一置为失效，再创建新会话。
- 原因：满足业务“单设备”要求，实现确定性最强。
- 备选方案：
  - 仅按 `device_id` 覆盖：允许多设备并存，不满足当前规则。

### 4) 验证码状态机：以数据库状态驱动一次性消费
- 决策：`sms_code_log` 记录状态（待验证/已验证/已过期/已失效）与错误次数；登录时仅允许消费最新可用记录一次。
- 原因：便于并发下保证“验证码不可重复使用”。
- 备选方案：
  - 全量 Redis 暂存验证码：速度快，但审计与留痕弱，不利于后续追踪。

### 5) 协议同意前置：登录请求携带协议确认信息并先校验
- 决策：验证码登录接口中显式提交同意信息（协议类型+版本+accept=true）；未满足时直接拒绝登录并不创建会话。
- 原因：满足合规前置约束，并可在首次登录前用手机号留痕。
- 备选方案：
  - 登录成功后补记同意：与“未同意禁止登录”冲突。

## Risks / Trade-offs

- [验证码并发校验导致重复消费] -> 在读取验证码记录时加事务与状态更新条件，保证仅一次成功更新。
- [短信供应商未接入影响联调] -> 提供短信发送网关接口与本地 stub，实现业务流程先行。
- [单设备策略引发用户多端体验下降] -> 后续可通过配置切换为“多设备”策略，本次先满足既定规则。
- [token 仅存数据库可能增加读压] -> 可选 Redis 做 session cache，数据库作为权威源；先实现数据库权威模型再优化。
- [协议版本来源暂未定] -> 登录服务先支持由配置项注入当前版本，后续可切到配置中心/数据库。

## Migration Plan

1. 在 `openspec` 完成认证能力 spec 与任务拆分。
2. 先编写测试（发送频控、验证码校验、会话恢复、全部退出、协议前置）。
3. 实现认证模块骨架（Controller/Service/Repository/Entity/DTO）。
4. 接入 Spring Security token 过滤与匿名白名单接口。
5. 本地联调通过后灰度上线；若异常可通过关闭认证入口或回滚应用版本恢复。
