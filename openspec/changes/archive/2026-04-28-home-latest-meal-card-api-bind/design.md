## Context

首页当前在 `HomePage` 中使用本地 mock 渲染“最近记录”卡片，无法体现用户真实最近一餐。后端 `GET /api/meal-records/latest` 已可返回最近已发生一餐（或 `data=null`），具备前端接入条件。

本次是纯客户端接入与交互规范化，不涉及后端接口改造。核心挑战在于：保证首页首屏稳定渲染的同时，引入“进入即请求”的实时数据；并处理空数据、失败降级、跳转记录页面三种状态。

## Goals / Non-Goals

**Goals:**
- 首页进入即发起最近一餐查询请求。
- 最近一餐存在时，卡片同时展示餐别与时间，并展示整数热量（`kcal`）。
- 无记录时显示“引导去记录一餐”并可跳转到记录页面。
- 请求失败时保持首页可用（降级占位，不影响其他区块）。

**Non-Goals:**
- 不调整后端接口结构与返回字段定义。
- 不改动今日热量卡片和 AI 建议区块的数据来源。
- 不在本次引入缓存、重试队列、离线持久化等复杂策略。

## Decisions

### Decision 1: 进入首页即请求，卡片局部刷新
- 方案：页面初次展示时触发请求，最近一餐区块单独维护 loading/data/error 状态。
- 理由：满足“进入即请求”的产品要求，同时避免整页等待接口阻塞。
- 备选：下拉刷新才请求（不满足当前要求）。

### Decision 2: 展示口径固定为“餐别 + 时间 + 整数热量”
- 方案：`mealType` 与 `recordedAt` 同时展示；`totalEstimatedCalories` 在 UI 层转为整数显示（四舍五入/截断由实现统一选择）。
- 理由：信息密度与可读性平衡，且符合用户确认的展示要求。
- 备选：仅展示餐别或保留两位小数（均不符合已确认规则）。

### Decision 3: 空态优先引导转化
- 方案：接口返回 `data=null` 时展示引导文案和可点击入口，点击跳转记录饮食页（`RoutePaths.recordMeal`）。
- 理由：空态不是错误，直接引导首餐记录更符合业务目标。
- 备选：展示纯空白占位（转化价值低）。

### Decision 4: 错误场景降级而非弹错
- 方案：请求失败时最近一餐区块显示稳定占位（可保留“查看全部”入口），不弹阻断式错误。
- 理由：首页是高频入口，稳定性优先。
- 备选：全局报错弹窗（打断用户流程）。

## Risks / Trade-offs

- [风险] 接口延迟导致卡片闪烁或跳变  
  → Mitigation：首屏使用固定骨架/占位高度，数据回来后再替换内容。

- [风险] 热量值为空或异常格式导致展示不一致  
  → Mitigation：UI 层做空值兜底（如 `-- kcal`）并统一整数化逻辑。

- [权衡] 进入即请求会增加一次首页网络开销  
  → Mitigation：仅请求单条最近记录，且失败不重试风暴，保持轻量。

## Migration Plan

1. 在 OpenSpec 中更新首页相关规范与任务清单。
2. 客户端实现最近一餐查询 model/repository/provider 与首页绑定。
3. 增加 widget/集成测试覆盖：有数据、无数据、失败降级、跳转行为。
4. 小流量验证首页稳定性后全量启用。
