一、第一条：项目治理规范（最重要）

文件路径：

openspec/specs/project-governance/spec.md

内容：

# 项目治理规范（project-governance）

本规范定义食迹项目的开发规则与规范驱动开发流程。

---

## 需求：规范驱动开发

系统必须采用规范驱动开发（Spec-driven development）。

### 场景：开发新功能

- 当开发者需要新增功能
- 必须先在 `openspec/changes/` 中创建变更提案
- 在提案通过后才可以进入实现阶段

---

## 需求：规范是唯一事实来源

系统行为以 OpenSpec 中的规范为准。

### 场景：代码与规范冲突

- 如果实现代码与规范不一致
- 必须修改代码使其符合规范
- 或者通过新的 change 修改规范

---

## 需求：所有文档必须使用中文

本项目所有规范文档必须使用中文。

### 场景：生成规范文档

- 当 AI 或开发者生成规范文档
- 必须使用中文编写
- 不允许生成英文规范

---

## 需求：AI生成代码必须可读

AI生成的代码必须具备可维护性。

### 场景：AI生成代码

- 代码必须符合项目目录结构
- 必须包含必要注释
- 不允许生成复杂但不可理解的实现

这一条的作用是：

约束 AI 的行为。

二、第二条：客户端架构规范（Flutter）

文件：

openspec/specs/client-architecture/spec.md

内容：

# 客户端架构规范（client-architecture）

本规范定义移动端（Flutter）的架构规则。

---

## 需求：客户端不得直接调用 AI 服务

客户端必须通过后端访问 AI 能力。

### 场景：用户触发 AI 分析

- 当客户端需要 AI 分析
- 必须调用后端 API
- 后端再调用 AI 模型服务

---

## 需求：客户端按功能模块组织代码

Flutter 项目必须采用 feature-first 结构。

### 场景：新增功能模块

- 当新增功能模块
- 必须在 `features` 目录中创建独立模块
- 页面、状态、模型必须放在同一模块内

---

## 需求：客户端不存储生产密钥

移动端不得保存生产环境 API key。

### 场景：AI接口调用

- API key 只能存在后端
- 客户端只调用后端服务

这一条会避免一个大坑：

Flutter 直接调用 Gemini API。

内容：

# 服务端架构规范（server-architecture）

本规范定义食迹后端服务架构规则。

---

## 需求：服务端负责业务编排

所有业务逻辑必须在后端实现。

### 场景：AI饮食分析

- 当用户提交饮食记录
- 后端负责数据校验
- 后端调用 AI 服务
- 后端返回分析结果

---

## 需求：服务端采用分层架构

后端必须采用标准分层结构。

### 场景：新增接口

- Controller 负责接收请求
- Service 负责业务逻辑
- Repository 负责数据库访问
- 外部服务调用必须封装在独立模块

---

## 需求：接口返回统一结构

所有 API 返回统一格式。

### 场景：接口返回数据

返回结构必须为：

{
"code": 0,
"message": "success",
"data": {}
}

四、第四条：AI能力规范

这个非常重要，因为你的项目是 AI 产品。

文件：

openspec/specs/ai-capability/spec.md

内容：

# AI能力规范（ai-capability）

本规范定义食迹项目中 AI 能力的基本原则。

---

## 需求：AI能力由后端统一管理

AI调用必须由后端统一管理。

### 场景：用户请求分析

- 客户端提交文本或图片
- 后端负责拼接 prompt
- 后端调用 AI API
- 后端返回分析结果

---

## 需求：AI提示词必须版本化管理

AI提示词必须存储在代码仓库中。

### 场景：新增 AI 功能

- prompt 必须写入 prompts 文件
- 不允许直接写死在代码中

---

## 需求：AI必须具备安全兜底

AI输出必须经过安全处理。

### 场景：AI返回异常内容

- 后端必须进行内容过滤
- 必须提供兜底提示
