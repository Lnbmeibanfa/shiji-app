# 食迹项目开发指南

本指南面向参与食迹（Shiji）项目的所有开发者，用于统一开发方式与协作流程。

---

## 一、项目开发原则

- **规范驱动开发（Spec-driven Development）**
  - 所有功能以 `openspec/specs/` 与 `openspec/changes/` 中的规范为唯一事实来源。
  - 没有 Spec / Change 的功能，不进入实现阶段。

- **测试驱动开发（TDD）**
  - 新功能必须先有测试，再写实现代码。
  - 测试应覆盖核心流程、边界情况和主要异常路径。

- **AI 辅助开发**
  - 优先使用 Cursor + OpenSpec 协作，让 AI 在明确边界内工作。
  - AI 生成代码必须可读、可维护，并同时生成必要测试。

- **所有文档必须中文**
  - `docs/`、`openspec/`、`README` 等文档一律使用中文编写。
  - 如需使用英文术语，应在中文上下文中给出简要解释。

---

## 二、OpenSpec 的使用规则

- **Spec 的作用**
  - 描述系统「应该如何工作」的长期规则与约束。
  - 主要存放在 `openspec/specs/`，用于定义架构、领域能力、AI 能力等。
  - 稳定性高，变更频率相对较低。

- **Change 的作用**
  - 描述某一次「具体改动」：要增加什么能力、修改什么行为、影响哪些场景。
  - 存放在 `openspec/changes/`，与一次迭代或一次 PR 强相关。
  - 用于记录需求背景、场景、输入输出和边界情况。

- **什么情况下必须先写 Change**
  - 新增任何业务功能（新页面、新接口、新 AI 能力等）。
  - 调整现有接口字段或语义。
  - 对 Spec 约定的行为进行例外处理或扩展。
  - 对现有模块做结构性重构，影响到外部调用方式。

> 总结：**有行为变更，就先写 Change；有长期规则变更，再更新 Spec。**

---

## 三、TDD 的使用规则

- **所有新增功能先写测试**
  - 在 Spec 与 Change 明确后，先根据业务场景设计测试用例。
  - 不允许直接从需求跳到实现，中间必须有测试。

- **后端优先写接口测试和服务测试**
  - 接口（Controller）测试：验证请求/响应结构、状态码和核心分支。
  - 服务（Service）测试：验证业务编排、领域规则和边界情况。
  - 必要时补充仓储（Repository）测试，确保关键查询/持久化逻辑正确。

- **前端优先写页面状态测试、组件测试和关键流程测试**
  - 页面状态测试：验证状态变化、加载/错误态、交互流程。
  - 组件测试：验证关键组件的渲染逻辑与交互行为。
  - 关键流程测试：如「记录一条饮食→发起 AI 分析→查看结果」等端到端流程。

- **测试通过后再补实现**
  - 先让测试失败（Red），再实现/修正代码到测试通过（Green），最后重构（Refactor）。
  - 重构时必须保持测试全部通过，确保行为未被意外修改。

## 四、Cursor 的使用方式

- **先让 Cursor 阅读 Spec 和 Change**
  - 在对话中明确指出相关的 `openspec/specs/*` 与 `openspec/changes/*` 文件。
  - 说明当前要修改的业务模块（如 `food_record`、`ai_analysis` 等）。

- **再让 Cursor 生成测试**
  - 先请求 Cursor 根据 Spec / Change 补齐或新建测试文件。
  - 测试应覆盖 PR 中计划实现的所有核心场景。

- **最后让 Cursor 生成实现**
  - 在测试框架稳定后，再让 Cursor 实现或修改对应模块代码。
  - 要求其遵守目录结构和模块边界（features/modules）。

- **不允许直接跳过测试写实现**
  - 禁止以「先写实现、后补测试」为默认路径。
  - 如确因调研或试验需要临时代码，应在合并前补齐 Spec / Change 和测试。

---

## 五、推荐开发流程

整体推荐流程（从需求到合并）：

1. **PRD**
   - 产品侧产出需求文档，描述业务目标与核心场景。
2. **OpenSpec Spec**
   - 在 `openspec/specs/` 中补充或更新相关领域/架构/AI 能力规范。
3. **Change**
   - 在 `openspec/changes/` 中创建本次迭代的变更文档，绑定具体场景与预期行为。
4. **测试用例**
   - 前端：在对应 `apps/mobile/test/features/<feature>/` 下编写测试。
   - 后端：在对应 `services/api/src/test/java/com/shiji/api/modules/<module>/` 下编写测试。
5. **实现代码**
   - 按前后端模块划分，在对应目录中实现代码。
   - 避免跨模块、跨层级的「临时穿透」写法。
6. **重构**
   - 在所有测试通过的前提下，整理内部结构与命名。
   - 如行为有调整，回到 Spec / Change 更新规范。
7. **合并**
   - 代码评审（Review）聚焦：是否符合 Spec / Change、测试是否充分、模块边界是否清晰。

## 六、前后端协作方式

- **接口字段以 Spec 为准**
  - 接口路径、请求/响应字段、错误码等统一以 OpenSpec 中的相关规范为准。
  - 前后端对齐时优先对齐 Spec，再对齐实现。

- **接口变更必须先修改 Spec**
  - 任何接口字段增减、语义变更、错误码调整，都必须先更新 Spec 与 Change。
  - 禁止只改代码不改规范，避免出现「文档与实现不一致」。

- **测试用例必须覆盖核心业务场景**
  - 前后端在对一个功能达成一致前，应先就测试用例达成一致。
  - 核心路径（主成功路径）和关键异常场景必须有自动化测试覆盖。

本指南与根目录 `README.md` 共同构成食迹项目的基础开发规范，后续如有调整，应通过 OpenSpec 的 Change 进行版本化记录。
