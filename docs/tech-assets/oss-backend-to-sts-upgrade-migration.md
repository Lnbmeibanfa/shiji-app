## 1. 背景与动机

### 1.1 技术动因

当前 v1 采用 **后端代理 PutObject**：文件字节经应用服务器再写入 OSS。该路径在餐图场景下实现快、权限集中，但随 DAU 与单量上升，**应用 ingress 带宽与连接数**将成为首要扩容约束，且与「业务逻辑算力」耦合。

### 1.2 升级目标

将 **C 端主路径**改为客户端持 **短期凭证** 直传 OSS，应用仅负责：鉴权、策略下发、元数据落库与可选完成校验，从而把大流量移出应用进程。

### 1.3 与现状关系

- **不迁移历史对象**：已存 OSS 对象与 `file_asset` 行保持不变。
- **表结构已预留**：`file_asset.upload_source` 支持 `client_direct`；`status` 支持 `pending` 等，可直接承接直传流程。

---

## 2. 目标与非目标

### 2.1 目标（成功标准）

| 编号 | 目标 | 验收方式 |
|------|------|----------|
| M1 | 新客户端主路径不经由 API 传输文件体 | 网关/应用层按字节 ingress 显著下降；抽样确认 |
| M2 | 资产记录仍统一落在 `file_asset` | 直传成功后记录可追溯，`fileId` 与业务关联方式与 v1 兼容或可映射 |
| M3 | 可灰度、可回退 | 特性开关关闭后回退代理上传；窗口内可观测 |

### 2.2 非目标

- 不改变现有已上传对象的 `object_key` 规则（除非单独变更）。
- 不在本阶段强制下线代理接口（可作为降级与旧客户端兼容）。
- 不替代异步内容安全扫描（仍可在后续流水线接入）。

---

## 3. 现状与目标态

### 3.1 现状（v1）

客户端 → `POST /api/files/upload`（multipart）→ 服务端 PutObject → `file_asset`（`upload_source=backend_proxy`，`status=uploaded`）。

### 3.2 目标态（v2）

客户端 → `POST /api/files/upload/credentials`（或等价）→ 返回 STS 临时凭证或 Post Policy → 客户端 **直传 OSS** → `POST /api/files/complete`（或 OSS 事件回调）→ 服务端校验并 `file_asset` 落库（`upload_source=client_direct`）。

### 3.3 对照表

| 维度 | v1 | v2 |
|------|----|----|
| 文件字节路径 | 经 API | 不经 API |
| 鉴权 | Session/JWT | 同左 + 临时凭证策略 |
| 首条 DB 记录时机 | OSS 成功后立即 `uploaded` | 可先 `pending` 再 `uploaded`（推荐） |
| 代理接口 | 唯一路径 | 降级/旧版保留 |

## 4. 影响范围

| 类型 | 对象 | 说明 |
|------|------|------|
| 后端 | `services/api` 文件模块 | 新增凭证、完成确认；RAM/STS 策略 |
| 客户端 | Flutter 等 | OSS SDK、重试、完成回调 |
| 运维 | 阿里云 RAM、OSS | 角色、策略模板、监控 |
| 数据 | `file_asset` | 新增写入路径；无需改表（已预留字段） |

---

## 5. 总体策略

**推荐策略**：**双路径并行 + 特性开关 + 客户端最低版本门槛**

**理由**：

1. 直传依赖客户端发版，无法与后端同日全量。
2. 凭证服务或网络异常时，**分钟级回退**代理路径保障业务连续性。
3. `object_key` 前缀与 v1 保持一致（`users/{userId}/meals/...`），便于 RAM 策略按前缀授权与生命周期规则复用。

**关键假设**：RAM 角色与 STS AssumeRole 已在预发验证；完成接口具备 **幂等**（建议 `clientUploadId` 或预分配 `fileId`）。

---

## 6. 详细步骤（建议分阶段）

### 阶段 A：准备

| 步骤 | 操作 | 验证 |
|------|------|------|
| A1 | 创建 RAM 角色；STS 策略限定 bucket、`users/*` 或更细前缀、允许的 OSS API（PutObject / 分片相关） | 预发仅可写测试前缀 |
| A2 | 实现「申请上传凭证」接口；返回过期时间、endpoint、临时 AK/SK、SecurityToken（或 Post 参数） | 单元/集成测试通过 |
| A3 | 实现「完成上传」：HeadObject 或事件驱动；`pending`→`uploaded`；写满 `etag`、`url` 等 | 与 v1 响应字段对齐 |

### 阶段 B：客户端改造

| 步骤 | 操作 | 验证 |
|------|------|------|
| B1 | 集成 OSS SDK，按凭证直传 | 预发端到端成功 |
| B2 | 失败重试、凭证过期重新申请 | 混沌场景通过 |

### 阶段 C：灰度与收敛

| 步骤 | 操作 | 验证 |
|------|------|------|
| C1 | 开关按用户比例或版本打开直传 | 监控错误率与延迟 |
| C2 | 缩容代理层资源（视流量曲线） | 成本与 SLO 达标 |
| C3 | 文档与 OpenSpec 归档合并 | 主 spec 更新 `file-upload` |

## 7. 数据与状态机

| 项 | 建议 |
|----|------|
| 直传首写 | 插入 `status=pending`，写入预分配 `object_key`（或由服务端生成后随凭证返回） |
| 完成 | Head 成功 → `uploaded`，`upload_source=client_direct` |
| 超时 | 定时任务扫描长期 `pending`，告警或清理（与 `expire_at` 字段配合） |

**无需历史数据搬迁**；v1 已 `uploaded` 的行保持不动。

---

## 8. 兼容与开关

| 开关 | 作用 | 回滚 |
|------|------|------|
| `upload.direct.enabled`（示例名） | 新客户端是否走直传主路径 | `false` 全走代理 |
| 客户端最低版本 | 低于则强制代理 | 下调或关直传 |

---

## 9. 风险与应急预案

| 场景 | 处置 |
|------|------|
| STS 策略错误导致 403 | 热修策略；未缓解前关灰度 |
| 凭证服务过载 | 扩容、限流；降级代理 |
| 完成确认丢失 | 幂等重试；对账任务修数据 |

---

## 10. 回滚方案

1. 关闭直传特性开关。
2. 客户端旧逻辑继续使用 `POST /api/files/upload`。
3. 已直传到 OSS 的对象保留；`file_asset` 以 DB 状态为准修复异常行。

**回滚验证**：抽样代理上传成功；无新增大量 `pending` 堆积。
